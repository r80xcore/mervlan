mervlan v0.53.07

Bug Fixes:

Native SSID passwords destroyed after every VLAN apply or heal (mervlan_manager.sh):
- After a VLAN apply completed, the native SSIDs (wl0/wl1) lost their Wi-Fi
  passwords and clients could no longer connect. VLAN SSIDs were unaffected.
  Resetting the password in the router UI restored connectivity until the next
  apply or heal triggered the issue again.
- Root cause 1: The manager was restarting the eapd authentication daemon bare
  (without firmware arguments) after restart_wireless had already started it
  correctly. On Broadcom platforms this reinitialises native radio authentication
  with an empty state, wiping the PSK that restart_wireless had just set up.
  Fix: eapd restart removed from the manager. restart_wireless handles it
  correctly on its own.
- Root cause 2: The unconfigured-interface sweep at the end of each apply briefly
  detached native radios (wl0, wl1) from br0 and immediately re-added them, even
  when they were already correctly placed. On Broadcom hardware this brief detach
  is enough to reset the authentication state for that radio.
  Fix: Native radios already in br0 are now left in place — the sweep skips the
  remove+reattach cycle when the interface is already where it belongs.

Shield gap on wireless events reduced (heal_event.sh, service-event-handler.sh):
- Wireless events had a fixed 3-second delay before heal_event.sh was launched.
  During that window the ebtables chains could be orphaned and clients left
  unguarded while the heal hadn't started yet.
- Fix: The 3-second delay is removed for wireless events — heal_event.sh now
  starts immediately. A DHCP gate rule (blocking UDP port 67) is installed in
  the inline re-link pass in service-event-handler.sh so no client can grab a
  br0 lease during the brief window before the heal takes over.
- To avoid acting on a stale healthy snapshot, heal_event.sh now polls for
  real evidence that the restart has begun (ebtables chains flushed or rc
  showing wireless activity) before running the main VLAN check. Shield restore
  runs on every poll tick. A 5-second ceiling prevents stalling if the restart
  is undetectable.

mervlan v0.53.06

Shield gap during service-event debounce delay (service-event-handler.sh):
- Between rc flushing ebtables and heal_event.sh's first protective tick there
  was a ~3-8s unprotected window: 3s MERV_HEAL_DELAY sleep + script startup.
  During this window both MERV_QT and MERV_MAC jump rules were orphaned and
  any associated client could reach br0 unguarded.
- Fix: service-event-handler.sh now runs an inline re-link pass synchronously
  for system events (restart_wireless and similar), after the debounce check
  passes but before the fire-and-forget subshell is launched. Only re-links
  existing chains' FORWARD/INPUT jump rules (~50ms); chain creation and DROP
  rule rebuild remain heal_event.sh's responsibility. This reduces the
  unprotected window from ~3-8s to ~50ms.

Trunk configuration delays placement detection by ~27s (heal_event.sh):
- On routers with a trunk port (e.g. RT-AX86U with eth5 trunk), VLAN bridge
  subinterfaces (eth5.X) keep br2/br3/br4/br5 structurally alive even after
  restart_wireless moves all wl interfaces back to br0. check_vlan_config's
  bridge-presence loop therefore passes 10/10, and the placement check that
  detects wl misplacement only ran at the tail — after the full 27s monitoring
  window. Total exposure: ~47s from event to heal trigger.
- Fix: check_vlan_config now runs check_wl_iface_placements once at entry,
  before the monitoring loop. If wl subinterfaces are already misplaced when
  called (after wait_for_rc_quiet confirms rc is done), it returns 1
  immediately without running the 27s window. For trunk-configured routers
  this reduces heal latency from ~47s to ~13s (dominated by wait_for_rc_quiet).

mervlan v0.53.04

Bug Fixes:

IoT/VLAN clients escaping to br0 during manager-triggered wireless restart:
- During a full VLAN apply, the manager restarts wireless services and then
  waits for the system to settle before rebinding interfaces. This wait loop
  had no shield enforcement, leaving a window where client traffic could reach
  br0 unguarded. The loop now re-arms the L2 shields on every tick so the
  window is closed within ~1 second of the restart completing.
- The explicit re-arm pass that fires once rc goes quiet was also moved earlier
  so it runs before the settle delay rather than after.

False "orphaned chain - re-linked" log spam on every cron tick:
- A pattern mismatch in the orphan detection check caused it to always report
  the MERV_QT and MERV_MAC chains as orphaned, even when the shields were
  healthy. The re-link path was idempotent so no actual damage was done, but
  every cron run produced noisy warning logs. Pattern corrected.

Shield blind spot during VLAN health monitoring window:
- Between each of the 10 monitoring passes in check_vlan_config there is a
  3-second pause. During that pause the shields were not being policed, so a
  second rc event arriving mid-window could flush the chains and leak traffic
  for the full 3 seconds undetected. The pause is now an active tick loop that
  restores both shields every 100ms, matching the same protection already in
  place during the rc-quiet wait.

Shield blind spot while waiting for a slow interface to appear:
- When a wireless subinterface takes a long time to appear after a wireless
  restart, the manager waits with exponential backoff (up to 63 seconds). No
  shield enforcement ran during this wait, so a late rc wakeup could flush the
  chains while the manager was simply sleeping. Each backoff sleep is now
  preceded by a full shield restore check.

mervlan v0.53.02

Bug Fixes:

sync_nodes.sh — Node Receives Auto-Injected TRUNK1 Even When All Trunks Are Zero:
- sync_nodes.sh unconditionally rewrote the Trunks section of settings.json
  before syncing to nodes, zeroing all trunks then forcing TRUNK1=1 with all
  pool VLANs tagged whenever the VLAN pool was non-empty. This ignored the
  user's explicit trunk configuration and produced a settings.json on the node
  that contradicted what was set in the UI.
- Fix: the auto-trunk injection is now gated on whether the main router itself
  has at least one active trunk (TRUNK1-8 = "1"). If all trunks are "0" on
  the main, the node receives the settings.json as-is with no rewrite. If any
  trunk is active on the main, the existing backhaul-trunk injection runs
  unchanged for backward compatibility.

MERV_MAC — Structural Fingerprint for JFFS and ebtables-Churn Decisions:
- merv_mac_merge_db used raw md5sum of the active db to decide whether to write
  the JFFS checkpoint. merv_mac_snapshot used raw md5sum to decide whether to
  rebuild ebtables rules. Because merv_mac_merge_db always refreshes timestamps
  for seen clients (latest-timestamp-wins), the raw content changed on every
  5-minute cron tick even when the same clients were on the same VLANs — causing
  JFFS writes and ebtables flush+apply on every tick on a stable network.
- Fix: both comparisons now use a structural fingerprint derived from the
  mac+iface+vid fields only (awk '{print $2,$3,$4}' | sort | md5sum). Timestamps
  are excluded. Result: a stable network produces zero JFFS writes and zero
  ebtables churn per cron tick. Writes and rebuilds occur only when the MAC set
  or their VLAN assignments actually change.

New Features:

MERV_MAC — Per-Client L2 Secondary Shield (Issue #22 mitigation):
- Added a second ebtables chain (MERV_MAC) that enforces client MAC → VLAN
  assignments independently of interface bridge placement. Specifically targets
  the post-watchdog restart_wireless race where firmware can complete a second
  wireless restart after heal_event.sh's monitoring window has closed, silently
  moving wl SSID subinterfaces back to br0 while VLAN bridges remain alive
  (trunk interfaces keep them up). In that window, previously-associated
  clients can obtain br0/native DHCP leases.
- MERV_MAC rule shape: -s <mac> --logical-in br0 -j DROP
  Rules self-dormant when the client's wl interface is in its correct VLAN
  bridge (--logical-in br0 only fires when enslaved to br0). Zero cleanup
  required after correct bridge placement.
- Persistent db: space-separated records (<epoch_ts> <mac> <iface> <vid>),
  active at /tmp/mervlan_tmp/mac_shield.db, checkpointed to JFFS at
  $MERV_BASE/tmp/mac_shield.db. Latest-per-MAC semantics; 48h max age prune.
  Boot restores from JFFS before the first apply (closes the reboot gap).
- wl assoclist snapshot: captured from each VLAN bridge after the post_rc_watchdog
  correction window closes (~WATCHDOG_DELAY_SEC + 5s). Precondition check
  confirms all expected wl interfaces are on their correct bridges before
  snapshotting to avoid polluting the db with mid-restart state.

check_wl_iface_placements() — Per-Interface Placement Detector:
- Replaces wl_leaked_to_br0 as the secondary leak detector in check_vlan_config.
  wl_leaked_to_br0 had a blind spot: its dual condition (wl in br0 AND none in
  VLAN bridge) goes false when a native guest SSID legitimately lives in br0 and
  at least one managed SSID is correctly placed. Partial leaks were invisible.
- New function uses merv_mac_build_expected_iface_vid (settings + NVRAM derived)
  to check each expected wl subinterface individually. Fires on ANY misplaced
  managed interface, regardless of guest SSID state. Falls back to wl_leaked_to_br0
  if lib_mervqt.sh is not loaded (graceful degradation on partial install).

MAC Shield Refresh Button (UI):

Performance:

heal_event.sh — Active Shield: Bootstrap Tick Probe (usleep/sleep Selection):
- At startup, heal_event.sh probes for usleep availability. When present, sets
  TICK_CMD="usleep 100000" and TICKS_PER_SEC=10 (100ms ticks per loop iteration).
  Falls back to TICK_CMD="sleep 1" / TICKS_PER_SEC=1 on firmware without usleep.
- wait_for_rc_quiet now runs at up to 10 ticks/sec on typical hardware: jump rules
  are re-linked within 100ms of each rc flush instead of up to 1s. Both TICK_CMD
  and TICKS_PER_SEC are exported so callers inherit the probed rate.

heal_event.sh — Active Shield: Single Shared ebtables Dump per Tick:
- Previously, restore_merv_qt_shield and restore_merv_mac_shield each issued
  their own ebtables -t filter -L call on every tick (4-6 netlink round-trips/tick).
- Fix: wait_for_rc_quiet fetches one shared dump per tick and passes it to both

  standalone. Reduces steady-state to 1 netlink call/tick.
- Also eliminates date +%s forks from the wait loop; timing is now a pure-tick

heal_event.sh / lib_mervqt.sh — Active Shield: Repair Log Deduplication:
- Both restore functions previously logged a repair message on every tick where
  damage was detected. At 10 ticks/sec over a ~90s restart_wireless event: ~900

  in the noise.
- Fix: Script-level _MERV_QT_SHIELD_STATE and _MERV_MAC_SHIELD_STATE track the
  current repair state per process invocation. Repair messages fire only on state
  transitions (first detection logs the damage type). A single recovery line fires
  when each shield transitions back to stable. Repair ticks in between are silent.
- Typical restart_wireless log: 1 "orphaned" line per shield on first detection,
  silence throughout the sustained damage window, optionally 1 "stable" line per
  shield when rc stops flushing. States: "" (initial) / "orphaned" / "wiped" / "ok".

lib_mervqt.sh — restore_merv_mac_shield: Split Repair Path (mirrors MERV_QT):
- Previously any damage (orphaned or wiped) triggered full flush → init → apply
  from db: 11+ ebtables ops plus a db file read on every affected repair tick.
- Fix: repair path now mirrors the two-case design already in restore_merv_qt_shield:
  (a) Orphaned (chain intact, FORWARD/INPUT jumps flushed): call ebt_mac_shield_init
      only. Chain and per-MAC DROP rules are untouched; only the 2 jump rules are
      re-linked. ~3 ebtables ops, no db read. This is the common rc flush pattern.
  (b) Wiped (chain gone): full flush → init → apply from db. Unchanged behavior.
- Reduces per-tick repair cost for the common orphaned case from 11+ ops to 3.
  ebt_mac_shield_init is idempotent: creates chain if missing, inserts jumps only
  when absent (grep guard), safe to call without a prior flush.

Bug Fixes:

heal_event.sh — Active Shield Regression: Radios Permanently Down After MAC Blacklist Change:
- v0.52.99 introduced evict_wl_from_br0() inside the wait_for_rc_quiet() loop
  as part of the Active Shield. This caused a regression where WiFi went down
  and did not recover without a manual apply.
- Root cause: evict_wl_from_br0() calls `wl -i IF down` + `brctl delif br0 IF`

  they inspect bridge structure, not radio state — so the heal completed without
  detecting the dead radios.
- Fix: removed evict_wl_from_br0() from the wait loop. Only restore_merv_qt_shield()
  remains in the loop. It is passive (ebtables DROP rules only, no wl or brctl
  calls) and does not interfere with firmware's restart sequence. The single
  evict_wl_from_br0() call after wait_for_rc_quiet() returns (once rc is
  confirmed quiet) is unaffected — that remains the correct clean sweep.

mervlan v0.52.98

Adds MERV_QT — a dedicated ebtables filter chain that provides true L2 quarantine
for managed wl subinterfaces while they are members of br0. Completes the DHCP
isolation work started in v0.52.97: where that release reduced the leakage window
to near-zero via the DHCP gate and pre-eviction, MERV_QT closes it at the bridge
layer. Any managed VAP temporarily in br0 (e.g. during a firmware restart_wireless)
is immediately dark at L2 — DHCP discovery, ARP, and static-IP forwarding are all
dropped. The --logical-in br0 constraint makes each rule automatically dormant once
the interface is correctly placed in its VLAN bridge, requiring no per-interface
cleanup for VLAN-bound ports.

New Features:

MERV_QT — L2 Quarantine Chain (mervlan_manager.sh):
- ebt_quarantine_init(): creates the MERV_QT chain in the ebtables filter table
  and inserts jump rules into FORWARD and INPUT. Idempotent: uses grep on -L
  output to detect existing jumps before inserting, safe to call multiple times.
- ebt_quarantine_flush(): flushes all per-interface rules from MERV_QT at the
  start of each apply cycle without touching the chain or jumps. Discards stale
  rules from the previous cycle before fresh ones are built from current NVRAM state.
- ebt_quarantine_add(): adds a -i IF --logical-in br0 -j DROP rule for a given
  wl subinterface. One rule covers both FORWARD and INPUT paths because both base
  chains jump to MERV_QT. Native base radios (wl0, wl1, …) are excluded. If
  --logical-in is unsupported by the firmware's ebtables binary, the add fails
  silently and the system degrades gracefully to evict-only behaviour (the
  heal_event.sh path already covers that case).
- ebt_quarantine_release(): removes the DROP rule for a specific interface.
  Called only in the none) case of attach_to_bridge after a successful brctl addif
  br0, lifting quarantine for interfaces intentionally placed in br0 (unmanaged
  SSIDs, native fallbacks). VLAN-bound interfaces never need this call — their
  rules go dormant automatically via the --logical-in br0 constraint.
- ebt_quarantine_teardown(): full chain removal in strict ebtables order: flush
  rules → delete FORWARD/INPUT jump refs → delete chain. Used directly by
  mervlan_boot.sh (inlined) and available for future callers in the manager.

cleanup_existing_config() Integration (mervlan_manager.sh):
- MERV_QT is armed as the very first action of every apply cycle, before any
  bridge teardown or trunk rule cleanup. Enumerates all wl*.* registered in NVRAM
  via grep on `nvram show` output, skipping native base radios. Any managed VAP
  that lands in br0 mid-cycle due to a firmware restart_wireless event is dropped
  at L2 immediately, closing the detection/confirmation gap that v0.52.97 narrowed
  but could not fully close.

setupdisable / nodedisable Teardown (mervlan_boot.sh):
- Both disable paths now run an inline MERV_QT teardown sequence (F → D FORWARD
  → D INPUT → X) before template removals. Chain is torn down silently with no
  restart_wireless required. Inlined rather than sourced because mervlan_boot.sh
  does not source mervlan_manager.sh.

mervlan v0.52.96

NOTABLE FEATURES:

restart_wireless Race Condition Fix (Security):
- Fixed a silent VLAN isolation failure where Asuswrt's restart_wireless event
  (triggered by MAC filter changes, channel changes, etc.) could leave guest
  and work VLAN wireless interfaces in br0, granting isolated clients full
  access to the internal LAN until a manual apply was forced.
- Root cause was two interacting flaws: trunk subinterfaces kept VLAN bridges
  structurally alive, causing the bridge-presence validator to report a false-
  positive healthy state even after all wl interfaces had been evicted; and
  wait_for_rc_quiet was only ever called after a mismatch was detected, so the
  firmware's delayed wireless reconstruction was never caught.
- Three layered fixes were applied (see heal_event.sh below).

Leak Detector — wl_leaked_to_br0():
- New dual-condition function that detects the specific failure mode where
  trunk subinterfaces keep VLAN bridges alive but wireless SSID subinterfaces
  (wl*.*, ra*.*, ath*.*) have been evicted back to br0.
- Only flags a leak when wl subinterfaces are present in br0 AND none are
  found in any non-br0 VLAN bridge. The dual condition self-filters unmanaged
  native Asuswrt guest networks (e.g. an unmanaged wl0.2 legitimately in br0
  won't trigger an alert as long as MerVLAN's own managed interfaces are
  correctly placed). Post-heal, once managed interfaces are restored, the
  check clears automatically — no settings cross-reference required and no
  infinite heal loop possible.
- Called as a secondary test at the tail of check_vlan_config, after all
  bridge-presence passes succeed.

Wireless Event Wait Ordering (Fix 2):
- wait_for_rc_quiet is now called before check_vlan_config for
  restart_wireless class events (max_wait raised to 120s to cover slow
  Broadcom DFS restarts on older hardware). Reading kernel bridge state
  mid-restart was producing false-positive healthy results because the
  firmware had not yet finished migrating wl interfaces.
- Non-wireless events (firewall, WAN, NAT, etc.) are unaffected.

Deferred Safety-Net Recheck (Fix 3):
- After any wireless event completes its primary monitoring window, a
  background subshell schedules a one-shot follow-up heal_event.sh call
  90 seconds later. This catches the edge case where firmware rc yields
  mid-restart (satisfying wait_for_rc_quiet) and then resumes and re-
  migrates wl interfaces after the monitoring window has already closed.
- The deferred event is routed through the full check_vlan_config and
  wl_leaked_to_br0 pipeline. The existing cooldown prevents a double-apply
  if the primary window already healed. Deferred events do not schedule
  further deferred events (no daisy-chaining).

- Robust SSID resolution: stronger trimming/normalization, guest slots widened to 1–9, auto/any uses all-band resolver with ambiguity warnings.
- Safer VLAN binding: clearer none/trunk behavior, stronger bridge membership verification (sysfs + brctl fallback), and post-restart second-pass + watchdog rebind.
- Safer operations: dry-run now avoids system changes (locks/state/changelog + command echoing), and service restarts use rc/proc busy detection + timeouts.
- AP isolation: immediate wl apply plus persistent mapping via ifname→nvram base key.
- JSON config parsing: delegated to shared lib_json helpers; supports legacy + nested Hardware layout.

heal_event.sh v0.47
heavily reworked to be better at finding and handling events that breaks
the VLAN bridges. Will do a 5-stage pass over 10 seconds when detecting 
a bridge compromising event. Is coupled with a CRU job that does a 1-pass
every 5 minutes silently. Will report every 12h. If any pass finds changes
inside the VLAN bridges, mervlan_manager.sh will be executed. Both heal_event
and mervlan_manager has locks, preventing hammering. Has a 90s cooldown.

service-event-handler.sh v0.47
Changed calls to heal-event to be run outside of the pipeline to prevent
that asuswrt-merlin always finishes last and queues heal_event.sh after
a VLAN compromising event is done, ensuring fast bridge healing.
